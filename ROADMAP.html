susurro — roadmap

Un gateway de voz propio. STT + TTS para todas las apps de Bernard, con llave solo suya. El vendor (Azure hoy, self-hosted después) es un detalle de implementación.

Verdad sin marketing: el backend está listo y es el activo real. El frontend web hasta ahora era un experimento de Matrix que no transcribía nada en producción. Este roadmap declara lo que susurro debe ser: una herramienta, no un demo.

Lo que susurro NO es (y nunca debió ser el SLA principal)

✗
Un "AI Transcription Studio" con Matrix rain y terminal de NORAD
Era un demo de Whisper in-browser de un pivot anterior. Cargaba 2.69 MB de ML para mostrar tres cards que decían "Not active". No era el producto.
✗
El experimento de Murmuraba / in-browser Whisper como feature central
Sin COOP/COEP en prod, SharedArrayBuffer era undefined — el motor de audio nunca arrancó en Azure. Era un cartel de funciones que nadie podía usar.
✗
Un competidor de Deepgram / ElevenLabs con features de ML propias
El valor es el GATEWAY propio: llave tuya, metering tuyo, motor intercambiable. No el motor en sí mismo (hoy Azure, mañana self-hosted). No hay que reinventar el STT/TTS.

Lo que la web SÍ debe ser

Una herramienta de tres páginas, sin cosplay de terminal:

/ → Onboarding y docs. Para que tus agentes (y cualquier dev) aprendan a usar susurro en 30 segundos. Token de onboarding visible, ejemplos curl/JS/Python copy-paste. Ya existe y funciona.

/admin → Dashboard de llaves: qué proyecto usa qué, cuánto consumen, crear/revocar. Ya existe y funciona.

/playground (pendiente) → Un micrograbador real: microfono → /v1/stt → muestra el texto. Cuadro de texto → /v1/tts → reproduce el audio. Usa el token de onboarding desde /v1/discovery, ningún hardcode. Sin ML in-browser, sin canvas, sin WASM. Funciona porque llama a tu gateway real como cualquier consumer.

El playground prueba el contrato del gateway, no el motor. Es la demostración honesta: si el gateway está vivo, el playground funciona. Si el motor cambia (Azure → self-hosted), el playground no se toca.

Contrato del gateway (estable, los consumers lo hornean)

GET /health Liveness — no requiere auth. Container Apps probe.
GET /ready Prueba real TTS de 1 char al upstream. Se pone 503 cuando el motor está caído.
GET /v1/discovery Machine-readable: devuelve el onboarding token + instrucciones de uso. Tus agentes lo leen para auto-onboardearse.
POST /v1/stt?language=es Audio bytes → { transcript, engine }. Bearer requerido. engine=whisper (default) o engine=deepgram.
POST /v1/tts { input, voice?, format? } → audio bytes (mp3 default). Bearer requerido.
POST /v1/refine { web_speech_text, deepgram_text } → { refined } — Claude elige la mejor transcripción.
GET /admin/keys Lista llaves + consumo. Admin Bearer requerido.
POST /admin/keys Crea llave. { name, daily_limit }. Admin Bearer requerido.
POST /admin/keys/:token/revoke Revoca una llave. Admin Bearer requerido.

P3 SOMEDAY — soberanía total

Self-hosted: cero vendor en la voz

◦
Reemplazar Azure whisper por faster-whisper en el container
El gateway ya abstrae el motor. Swappear whisper: agregar faster-whisper al Dockerfile, un endpoint /v1/stt que lo use internamente, apagar AZURE_WHISPER_DEPLOYMENT. El contrato no cambia. Costo: cero por transcripción, a cambio de CPU del container.
◦
Reemplazar Azure tts por Kokoro/Piper
TTS open source de calidad decente. Mismo patrón: en el container, swappear el motor, contrato /v1/tts no cambia. Consumers no se enteran.
◦
Una vez verde el P3: borrar susurro-openai
La cuenta, el recurso y los deployments tts+whisper desaparecen. Voz 100% soberana, costo de infra solo. Eso es lo que dijiste que querías desde el principio.

El SLA que importa

El SLA de susurro no es "el demo web se ve bonito". Es:

1. El discord-bot habla. (P1 — roto hoy)

2. Videopipe genera voiceovers. (P1 — roto hoy)

3. Cualquier nueva app que Bernard arranque usa susurro desde el día 1, con su llave, sin saber qué vendor está abajo.

4. /ready dice la verdad sobre si el motor sirve.

El web app es la vitrina y la herramienta de gestión. No es el producto. El producto es el gateway. Cuando el gateway está vivo, susurro cumple su función aunque nadie abra el browser.
